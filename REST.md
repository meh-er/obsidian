#DAWP 

REpresentational State Transfer
- API or Application Programming Interface with:
	- A base URL, such as `http://example.com/api`
	- Standard HTTP methods (e.g. GET, POST, PUT and DELETE)
	- A media type that defines data elements (JSON/XML)
- REST API+Documentation: skeleton generated by JHipster from JDL

### Constraints for RESTful
- Uniform Interface - design conventions
- Stateless
- Cacheable
- Client-server
- Layered System
- (Code on demand)

### Design Conventions

| **Task**             | **Method** | **Incorrect**    | **Correct**       |
| -------------------- | ---------- | ---------------- | ----------------- |
| list all todo notes  | `GET`      | /GetAllTodos     | /todos            |
| Create a todo        | `POST`     | /createTodos     | /todos            |
| Delete a todo note   | `DELETE`   | /deleteTodos     | /todos/145        |
| Get a todo's subject | `GET`      | /getTodosSubject | /todos/10/subject |

#### Plural not singular
- REST rarely used for single unique resources.
- Keep it simple: use only plural for resources.
- e.g. `/todos` instead of `/todo`

#### GET is stateless
Use `PUT`, `POST` and `DELETE` methods to change state
`GET` should never change the state of an item
INCORRECT:  `GET /items/923?publish` (don't use query parameters for REST operations)
CORRECT: `POST /items/923/publish`

#### Specify the format (JSON. XML, HTML) in the headers
`Content-Type`: defines the **request** format
`Accept`: defines a list of acceptable **response** formats

#### Object relations
Use sub-resources to get related resources
`GET /todos/923/items/4`
(from todo #923, get item #4 only)
